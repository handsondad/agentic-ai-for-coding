# AI for Coding 基础设施可执行优化蓝图

## 1. 目标与范围

本蓝图用于把当前仓库从“可运行的自动化框架”升级为“可规模化复用的 AI for Coding 基础设施”。

目标是让单个 Issue 默认可自动完成：

1. 需求理解
2. 代码实现
3. 单元测试与质量门禁
4. 提交与 PR 创建
5. 人工仅做最终 PR 审核

本蓝图聚焦工程基础设施，不覆盖具体业务功能开发。

## 2. 当前基础与差距

### 已有基础

1. Issue 到 worktree 到 prompt 的自动准备链路
2. 本地 backlog 与 GitHub issue 联动机制
3. 单 issue 与批量调度双模式
4. PR 模板和执行规范已建立
5. Python 3.12、ruff、pytest 等基础质量工具已接入

### 关键差距

1. 评估体系未形成闭环：缺少离线评测集与在线指标仪表
2. 测试门禁策略不够分层：缺少按风险分级执行策略
3. 失败分类与自动修复建议未标准化
4. 质量与效率指标缺少统一归档和趋势追踪
5. 多项目复用入口未明确为一套 Starter Kit

## 3. 目标架构

### A. 任务编排层

1. 输入标准化：Issue 模板、标签、验收标准结构统一
2. 任务切片：单 issue 默认串行，批量模式按优先级并发
3. 上下文组装：自动读取关联文档、历史 PR、模块依赖

### B. 执行与隔离层

1. 每个 issue 独立 worktree
2. 环境初始化 hooks 跨平台一致
3. 失败可重试与可回滚

### C. 质量门禁层

1. 快速门禁：format/lint/type
2. 逻辑门禁：unit test
3. 风险门禁：integration test/security scan
4. PR 描述门禁：模板章节完整性检查

### D. 评估与可观测层

1. 离线评估：固定 issue 数据集回放
2. 在线评估：真实执行成功率、返工率、Lead Time
3. 成本指标：token、执行时长、重试次数
4. 失败分类：需求缺口、代码缺陷、环境阻塞、外部依赖

## 4. 90 天执行路线图

## 阶段 1（第 1-2 周）：稳定性与门禁最小闭环

目标：保证单 issue 全流程稳定通过，并可追踪失败点。

交付：

1. 建立统一门禁入口脚本（统一执行 lint/test/type/checklist）
2. 引入失败分类输出规范（失败步骤、摘要、下一步建议）
3. 增加 PR 模板完整性校验脚本
4. 在 README 与 WORKFLOW 中固化默认执行顺序

完成标准：

1. 连续 10 个 task issue 自动执行成功率 >= 80%
2. 失败 case 100% 带可执行修复建议

## 阶段 2（第 3-6 周）：评估体系落地

目标：让优化决策由指标驱动，而不是凭经验。

交付：

1. 构建离线评测集（至少 20 条 issue 样本）
2. 设计评估维度：正确性、质量、流程完整性、效率
3. 输出评估报告模板（每次策略变更必须回放）
4. 记录线上指标快照（每周）

完成标准：

1. 每周产出一次评估报告
2. 关键指标可对比上周变化趋势

## 阶段 3（第 7-12 周）：平台化与多项目复用

目标：沉淀成可复制的 Starter Kit。

交付：

1. 新增 AGENTS.md 作为统一入口
2. 提供项目接入清单（30 分钟完成初始化）
3. 抽象配置项与脚本模板，支持新仓库复用
4. 增加治理节奏（周报/月报）

完成标准：

1. 新项目可在 1 天内接入全流程
2. 复用项目首次 2 周自动执行成功率 >= 70%

## 5. 可执行任务包（直接建 backlog）

建议按以下 task 拆分并创建本地 backlog 文件。

1. 统一门禁执行器
2. 失败分类与修复建议标准
3. PR 模板完整性校验器
4. 离线评估集与回放脚本
5. 在线指标采集与周报模板
6. Starter Kit 接入清单与模板化配置

每个任务都应包含：

1. 背景与动机
2. 需要做与不在范围
3. 可测试的完成标准
4. 最小可运行命令
5. 风险与回滚方案

## 6. KPI 与 SLO

核心 KPI：

1. 自动执行成功率
2. PR 一次性通过率
3. 平均执行时长（Issue 到 PR）
4. 人工返工率（PR 评论后需要重做的比例）
5. 测试门禁通过率

建议 SLO（初始版）：

1. 自动执行成功率 >= 80%
2. PR 模板完整率 = 100%
3. 单 issue 平均执行时长 <= 90 分钟
4. 单测通过率 >= 95%

## 7. 治理机制

周节奏：

1. 每周一次失败复盘（Top 3 阻塞原因）
2. 每周一次策略优化（只改一个关键变量）
3. 每周一次指标快照归档

月节奏：

1. 评估样本库扩充
2. 规范与模板升级
3. 对外复用能力评审

## 8. 项目启动并行策略

当启动新业务项目时，必须并行推进两条线：

1. 业务线：功能开发与需求交付
2. 平台线：门禁、评估、可观测、模板治理

Definition of Done 必须同时满足：

1. 功能实现完成
2. 质量门禁通过
3. PR 描述完整
4. 指标可记录
5. 可回放复盘

## 9. 立即执行清单（本周）

1. 创建 AGENTS.md 任务并排到高优先级
2. 新增统一门禁执行脚本任务
3. 新增离线评估集任务
4. 新增 PR 模板校验任务
5. 建立每周指标记录文件

执行顺序建议：

先稳定单 issue 成功率，再扩展评估体系，最后做多项目复用。
