# ADR-0001: 使用架构决策记录（ADR）追踪重要技术决策

**状态**：已接受  
**日期**：2025-01-01  
**决策者**：Tech Lead

---

## 背景

随着项目演进，团队成员需要了解为什么系统是现在这个样子。没有记录的技术决策会导致：
- 新成员不理解设计意图，做出与架构不符的修改
- 重复踩已知的坑
- AI Agent 生成的代码无法理解架构约束

## 决策

在 `docs/adr/` 目录中使用架构决策记录（ADR）追踪所有重要的技术决策。

**ADR 适用场景**：
- 技术选型（框架、数据库、库）
- 架构模式选择
- 影响多个模块的设计决策
- 有重大权衡取舍的决策

**ADR 不适用**：
- 小的实现细节
- 可以通过代码注释说明的决策

## ADR 格式

每个 ADR 是一个 Markdown 文件，命名格式：`XXXX-short-title.md`

```markdown
# ADR-XXXX: 标题

**状态**：[草案 | 已接受 | 已废弃 | 已替代]
**日期**：YYYY-MM-DD
**决策者**：[人名或团队]

## 背景
[为什么需要做这个决策？当前面临什么问题？]

## 决策
[我们决定做什么？]

## 理由
[为什么选择这个方案？]

## 备选方案
[考虑了哪些其他方案？为什么没有选择？]

## 影响
[这个决策带来了哪些正面和负面影响？]
```

## 理由

- ADR 轻量级，不需要特殊工具，Markdown 文件即可
- 与代码共同版本控制，可以通过 Git 历史追溯决策背景
- AI Agent 可以读取 ADR 来理解架构约束，避免生成不符合架构的代码

## 影响

**正面**：
- 新成员快速理解系统设计意图
- AI Agent 生成的代码更符合架构规范
- 减少架构退化风险

**负面**：
- 需要花时间编写 ADR（预计每个 ADR 15-30 分钟）
- 需要团队纪律保持 ADR 的更新
