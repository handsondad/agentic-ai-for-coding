## 🏗️ 1. 系统核心架构与任务流 (Canvas)
为了保证 AI 改代码时不会互相冲突，每个 Celery Worker 领到任务后，都会在本地创建一个完全隔离的 Git 工作区 (Git Worktree)。
Celery 官方推荐使用 Canvas (工作流拓扑) 来串联复杂任务。我们这里使用 chain（串行链），确保前一步成功后才走下一步：

【 监控器 (Cron / Beat) 】 -> 发现 Issue -> 触发 Celery 任务链 (Chain)
                                            │
    ┌───────────────────────────────────────┴────────────────────────────────────────┐
    ▼ (步骤 1: 环境隔离)             ▼ (步骤 2: Sub-agent 执行)             ▼ (步骤 3: 自动化测试)          ▼ (步骤 4: 提 PR)
┌──────────────────────┐       ┌───────────────────────────┐       ┌────────────────────┐       ┌──────────────┐
│  setup_workspace()   │ ───>  │     run_copilot_agent()   │ ───>  │    run_tests()     │ ───>  │  create_pr() │
│  创建独立 Git Worktree│       │  调用 Copilot 自动改代码    │       │ 运行 pytest/npm test│       │ gh pr create │
└──────────────────────┘       └───────────────────────────┘       └────────────────────┘       └──────────────┘

import osimport shutilimport subprocessfrom celery import Celery, chain
# 1. 初始化 Celeryapp = Celery('air_agent', 
             broker='redis://localhost:6379/0',
             backend='redis://localhost:6379/1')
REPO_PATH = "/path/to/your/main/project"  # 你本地的主项目绝对路径WORKSPACE_BASE = "/tmp/air_workspaces"      # 存放各 Sub-agent 隔离工作区的目录
# ---------------------------------------------------------# 步骤 1：为 Sub-agent 准备完全隔离的 Git 工作区# ---------------------------------------------------------
@app.taskdef setup_workspace(issue_id):
    branch_name = f"air-fix-issue-{issue_id}"
    workspace_path = os.path.join(WORKSPACE_BASE, f"issue_{issue_id}")
    
    # 清理历史残留
    if os.path.exists(workspace_path):
        shutil.rmtree(workspace_path)
        
    print(f"[Issue #{issue_id}] 正在创建隔离工作区...")
    # 使用 git worktree 在独立目录中切出新分支，绝不污染你当前正在写的代码
    subprocess.run(["git", "worktree", "add", "-b", branch_name, workspace_path, "main"], cwd=REPO_PATH, check=True)
    
    return {"issue_id": issue_id, "workspace_path": workspace_path, "branch_name": branch_name}
# ---------------------------------------------------------# 步骤 2：调用 GitHub Copilot / CLI 自动执行修改# ---------------------------------------------------------
@app.taskdef run_copilot_agent(workspace_info, issue_title, issue_body):
    workspace_path = workspace_info["workspace_path"]
    issue_id = workspace_info["issue_id"]
    
    print(f"[Issue #{issue_id}] Sub-agent 正在投入工作...")
    
    # 构造喂给 Copilot CLI 的深度 Prompt
    prompt = f"""
    根据以下 GitHub Issue 的要求，修改当前项目中的代码。
    【Issue 标题】: {issue_title}
    【Issue 描述】: {issue_body}
    
    请定位相关文件并直接进行修改。修改完成后，请确保代码符合项目原有的编码风格。
    """
    
    # 调用 gh copilot 执行任务。--allow-all-tools 允许 AI 自主读写本地文件
    # -p 代表以非交互模式（Headless）运行，无需人工敲回车确认
    cmd = ["gh", "copilot", "task", "-p", prompt, "--allow-all-tools"]
    
    # 在隔离的临时工作区目录中执行命令
    result = subprocess.run(cmd, cwd=workspace_path, capture_output=True, text=True)
    
    if result.returncode != 0:
        raise Exception(f"AI 编程失败: {result.stderr}")
        
    print(f"[Issue #{issue_id}] AI 代码修改完成。")
    return workspace_info
# ---------------------------------------------------------# 步骤 3：自动化回归测试（质量控制 Gate）# ---------------------------------------------------------
@app.taskdef run_tests(workspace_info):
    workspace_path = workspace_info["workspace_path"]
    issue_id = workspace_info["issue_id"]
    
    print(f"[Issue #{issue_id}] 正在运行自动化测试...")
    
    # 根据你的项目调整测试命令，比如 ["npm", "test"] 或 ["pytest"]
    test_cmd = ["pytest"] 
    result = subprocess.run(test_cmd, cwd=workspace_path, capture_output=True, text=True)
    
    if result.returncode != 0:
        # 如果测试失败，抛出异常。Celery 链会在这里断开，不会错误地去提 PR
        raise Exception(f"❌ 测试未通过！报错信息:\n{result.stdout}\n{result.stderr}")
        
    print(f"[Issue #{issue_id}] ✅ 测试全部通过！")
    return workspace_info
# ---------------------------------------------------------# 步骤 4：自动提交并创建 GitHub PR# ---------------------------------------------------------
@app.taskdef create_pr(workspace_info):
    workspace_path = workspace_info["workspace_path"]
    issue_id = workspace_info["issue_id"]
    branch_name = workspace_info["branch_name"]
    
    print(f"[Issue #{issue_id}] 正在推送代码并创建 Pull Request...")
    
    # 1. 提交代码
    subprocess.run(["git", "add", "."], cwd=workspace_path, check=True)
    subprocess.run(["git", "commit", "-m", f"fix: auto-resolved issue #{issue_id}"], cwd=workspace_path, check=True)
    
    # 2. 推送到远端仓库
    subprocess.run(["git", "push", "origin", branch_name], cwd=workspace_path, check=True)
    
    # 3. 使用 GitHub CLI 提 PR，并绑定原 Issue 自动关闭（Closes #X）
    pr_cmd = [
        "gh", "pr", "create",
        "--title", f"🤖 Auto-fix: Resolve Issue #{issue_id}",
        "--body", f"This PR is automatically generated by AIR Agent.\n\nCloses #{issue_id}",
        "--base", "main",
        "--head", branch_name
    ]
    subprocess.run(pr_cmd, cwd=workspace_path, check=True)
    
    # 4. 清理创建的 git worktree 空间
    subprocess.run(["git", "worktree", "remove", workspace_path], cwd=REPO_PATH, check=True)
    
    print(f"[Issue #{issue_id}] 🚀 PR 创建成功！任务圆满完成。")

import subprocessimport jsonfrom tasks import setup_workspace, run_copilot_agent, run_tests, create_pr, chain
def fetch_and_dispatch_issues():
    # 使用 GitHub CLI 获取最新的 5 个待处理 Issue
    cmd = ["gh", "issue", "list", "--state", "open", "--limit", "5", "--json", "number,title,body"]
    result = subprocess.run(cmd, capture_output=True, text=True)
    issues = json.loads(result.stdout)
    
    for issue in issues:
        issue_id = issue["number"]
        title = issue["title"]
        body = issue["body"]
        
        print(f"发现待处理任务 #{issue_id}，正在组装 Celery 任务链...")
        
        # 组装 Celery 串行链：创建空间 -> AI改代码 -> 跑测试 -> 提PR
        task_chain = chain(
            setup_workspace.s(issue_id),
            run_copilot_agent.s(title, body),
            run_tests.s(),
            create_pr.s()
        )
        
        # 异步推入消息队列，Worker 会立刻并行认领
        task_chain.delay()
if __name__ == "__main__":
    fetch_and_dispatch_issues()

## 🌟 这套细化方案的厉害之处

* 并行而不冲突：多亏了 git worktree，即使 Celery 同时启动 4 个 Worker 去改 4 个不同的 Issue，它们在本地都有独立的文件夹，绝对不会发生本地文件冲突或污染。
* 高容错率：如果 AI 改出的代码有语法错误，步骤 3 的 pytest 就会挂掉，Celery 会立刻中断该 Issue 的后续执行，不会把垃圾代码提到 GitHub 上。你只需要去 Celery 结果里看一眼报错日志即可。

用户问题：【有个问题，我的issue 可能是有依赖的，所以在调度执行的时候，要按照依赖关系去执行，celery可以实现这个功能吗】

AI回答：【可以，Celery 完全有能力处理带有依赖关系（有向无环图 DAG）的任务调度。
当你的 GitHub Issues 存在依赖关系（例如：必须先解决 #10 基础用户模块，才能解决 #11 支付模块）时，我们不能再简单地使用并发，而必须引入拓扑排序（Topological Sort）来动态构建 Celery 的任务图。
在 Celery 中，实现这种复杂依赖主要有两种最佳实践路径：一种是利用自带的高级工作流组件（Canvas 进阶版），另一种是更灵活、也是业界更推荐的“动态图生成”方案。
------------------------------
## 🛠️ 方案 A：利用 Celery 官方内置组件组装（Canvas 进阶）
Celery 的 chord（和弦） 和 group（组） 允许我们将依赖关系进行层级打包：

* 如果 Issue B 依赖 Issue A ➡️ 使用 chain(A, B) 串行。
* 如果 Issue C 依赖 Issue A 和 Issue B ➡️ 使用 chord(group(A, B), C)，即 A 和 B 并行，都成功后触发 C。

缺点：当依赖网络非常错综复杂时，用 Celery 自带的语法硬编码去嵌套 chord 和 group 会极其痛苦，代码很难维护。
------------------------------
## 🚀 方案 B：使用 Python 拓扑排序 + Celery 链式动态调度（最推荐）
这是解决 Issue 依赖最优雅、扩展性最好的方案：

   1. 在 GitHub Issue 中标记依赖：在 Issue 的描述（Body）里用特定格式写明依赖，例如 Depends on: #10。
   2. 构建依赖图：使用 Python 内置的 graphlib（Python 3.9+ 官方自带）对所有待处理的 Issue 进行拓扑排序。
   3. 动态串联 Celery 链：将排序后的 Issue 队列动态组合成一个多层级的 Celery 工作流。

## 💻 核心代码改造实现## 1. 从 Issue 描述中提取依赖关系
我们可以修改 main.py 的抓取逻辑，让它不仅读取 Issue，还能解析出它依赖哪一个 Issue：

import reimport jsonimport subprocessfrom graphlib import TopologicalSorterfrom tasks import setup_workspace, run_copilot_agent, run_tests, create_pr, chain, group
def fetch_issues_with_dependencies():
    # 抓取所有开启的 Issue
    cmd = ["gh", "issue", "list", "--state", "open", "--limit", "20", "--json", "number,title,body"]
    result = subprocess.run(cmd, capture_output=True, text=True)
    issues = json.loads(result.stdout)
    
    issue_dict = {}
    dependency_graph = {}
    
    for item in issues:
        issue_id = item["number"]
        issue_dict[issue_id] = item
        
        # 使用正则解析描述中的 "Depends on: #10"
        match = re.search(r"Depends on:\s*#(\d+)", item["body"], re.IGNORECASE)
        if match:
            parent_id = int(match.group(1))
            # 只有当父 Issue 也在待处理列表中时，才加入依赖图
            dependency_graph[issue_id] = {parent_id}
        else:
            dependency_graph[issue_id] = set() # 无依赖
            
    return issue_dict, dependency_graph

## 2. 使用拓扑排序动态提交 Celery 任务
我们把一个 Issue 完整的“隔离-AI修改-测试-提PR”封装成一个子流水线（Sub-Pipeline），然后根据拓扑排序的结果进行串联：

def build_issue_pipeline(issue):
    """为单个 Issue 生成它专属的 Celery 执行链"""
    return chain(
        setup_workspace.s(issue["number"]),
        run_copilot_agent.s(issue["title"], issue["body"]),
        run_tests.s(),
        create_pr.s()
    )
def dispatch_with_dependencies():
    issue_dict, dependency_graph = fetch_issues_with_dependencies()
    
    # 1. 使用 Python 官方的拓扑排序器
    ts = TopologicalSorter(dependency_graph)
    
    try:
        # ts.static_order() 会返回一个安全的执行顺序列表，例如: [10, 12, 11]
        # 10 无依赖先执行，12 其次，11 依赖 10 最后执行
        execution_order = list(ts.static_order())
    except ValueError:
        print("❌ 发现循环依赖（死锁）！请检查您的 GitHub Issues 依赖设置。")
        return

    print(f"🧬 成功计算出最优执行拓扑链: {execution_order}")
    
    # 2. 动态拼装最终的 Celery 顶级链
    final_workflow = None
    
    for issue_id in execution_order:
        if issue_id not in issue_dict:
            continue # 如果依赖的父 Issue 已经关闭了，直接跳过
            
        issue_data = issue_dict[issue_id]
        current_pipeline = build_issue_pipeline(issue_data)
        
        if final_workflow is None:
            final_workflow = current_pipeline
        else:
            # 将新的 Issue 任务链挂在上一阶段的屁股后面
            final_workflow = chain(final_workflow, current_pipeline)
            
    # 3. 一键推入 Celery 队列异步执行
    if final_workflow:
        final_workflow.delay()
        print("🚀 依赖关联流已成功推入 Celery 队列，正在按顺序安全消费...")
if __name__ == "__main__":
    dispatch_with_dependencies()

## 🛡️ 高级考量：如果上游依赖任务挂了怎么办？
在带有依赖的自动化系统中，最怕 Issue #10（底层核心）因为 AI 没写对导致测试失败挂掉了，结果下游的 Issue #11 依然硬着头皮去执行，从而引发连环报错。
由于我们使用了 Celery 的 chain：

* Celery 拥有天然的断链机制（Chain Breaking）。
* 如果整个长链中的任何一个任务（比如 #10 的 run_tests）抛出了异常，整条链后续的所有任务都会自动转为 REVOKED（取消）状态。
* 这样就完美确保了：只要底层的代码没修好，上层的 AI 绝对不会盲目跟进，保障了你私有仓库代码分支的绝对安全。

【单个 Issue 内部微观流】

       │

       ▼
 2. verify_test_fails (校验网关)  ───> 尝试跑一次测试。此时因为业务代码还没写，测试【必须失败】(Red)

       │
       ▼
 create_pr (提 PR)

## 💻 改造后的 Celery 任务代码 (tasks.py)
我们可以通过拆分任务，让不同的 Sub-agent 各司其职，并通过本地文件传递上下文：

# ---------------------------------------------------------# 步骤 1.1：测试智能体 - 只负责写测试用例# ---------------------------------------------------------
@app.taskdef write_test_agent(workspace_info, issue_title, issue_body):
    workspace_path = workspace_info["workspace_path"]
    issue_id = workspace_info["issue_id"]
    
    # 强制 AI 只能在测试目录活动，防止它偷懒把业务代码一起写了
    prompt = f"""
    你是资深测试专家。请根据以下 GitHub Issue 的需求，在 `tests/` 目录下编写完整的自动化测试用例（使用 pytest）。
    【Issue 标题】: {issue_title}
    【Issue 描述】: {issue_body}
    
    【铁律】：你只能在 `tests/` 目录下新建或修改测试文件，绝对不能修改 `src/` 或业务逻辑代码！
    """
    cmd = ["gh", "copilot", "task", "-p", prompt, "--allow-all-tools"]
    subprocess.run(cmd, cwd=workspace_path, check=True)
    
    print(f"[Issue #{issue_id}] 🤖 测试智能体：测试用例编写完成。")
    return workspace_info
# ---------------------------------------------------------# 步骤 1.2：验证网关 - 确保测试写对了（必须能捕获未实现的功能）# ---------------------------------------------------------
@app.taskdef verify_test_fails(workspace_info):
    workspace_path = workspace_info["workspace_path"]
    
    # 跑一下刚写的测试
    result = subprocess.run(["pytest"], cwd=workspace_path, capture_output=True, text=True)
    
    # 【核心逻辑】：如果此时测试直接通过了，说明 AI 写的测试是无效的（空测试），或者功能早就实现了
    if result.returncode == 0:
        raise Exception("❌ 异常：新编写的测试用例在没有修改业务代码的情况下竟然通过了！测试可能无效。")
        
    print(f"[Issue #{workspace_info['issue_id']}] 🎯 验证成功：测试成功失败（红灯），准备交由编码智能体实现业务。")
    return workspace_info
# ---------------------------------------------------------# 步骤 2：编码智能体 - 负责看图说话，让测试通过# ---------------------------------------------------------
@app.taskdef write_code_agent(workspace_info, issue_title, issue_body):
    workspace_path = workspace_info["workspace_path"]
    issue_id = workspace_info["issue_id"]
    
    # 让编码 AI 结合 Issue 描述和刚才写好的测试文件，去补全业务代码
    prompt = f"""
    你是资深后端开发工程师。请根据以下 GitHub Issue 的要求，修改项目中的业务代码（`src/` 目录），使得 `tests/` 目录下的测试用例能够完全通过。
    【Issue 标题】: {issue_title}
    【Issue 描述】: {issue_body}
    
    【核心任务】：请阅读当前项目中现有的测试用例，根据测试期望的输入输出来编写/重构业务逻辑。
    """
    cmd = ["gh", "copilot", "task", "-p", prompt, "--allow-all-tools"]
    subprocess.run(cmd, cwd=workspace_path, check=True)
    
    print(f"[Issue #{issue_id}] 🤖 编码智能体：业务代码修改完成。")
    return workspace_info

## ⛓️ 宏观调度与微观 Pipeline 的完美结合 (main.py)
现在，我们把这套“写测试 ➡️ 变红 ➡️ 写代码 ➡️ 变绿”的微观 Pipeline，组装到你之前的大拓扑依赖图中。
在 main.py 中，为单个 Issue 生成的 Pipeline（子链）应该更新为：

def build_issue_pipeline(issue):
    """
    单个 Issue 的微观闭环：

## 🛠️ AI + 人工协同开发规范 (AIR-SPEC-v1)
本规范旨在定义一个由人类战略驱动、规范（Spec）作为契约、AI Agent 自动化闭环交付的软件开发生命周期模型。
------------------------------
## 1. 核心角色与协作边界 (Who & What)
在整个开发周期中，人类与 AI 分工明确，遵循“人类做主观决策、AI 做客观执行”的原则。

* 人类 PM / 架构师 (Human Link)：负责定义产品源头（PRD）和顶层架构。不写具体代码，只负责审查 AI 生成的“契约（Spec）”与 PR。
* 拆单机器人 (Orchestrator Agent)：负责理解人类的 PRD，并将其严丝合缝地拆解为具有依赖关系的标准化 GitHub Issues。
* 开发流脚手架 (AIR Sub-agent Pipeline)：基于 Python + Celery + GitHub CLI/Copilot 的底层调度器。负责拉取 Issue、派发子 Agent、环境隔离、自动化测试与 PR 提交。

------------------------------
## 2. 宏观开发工作流规范 (Workflow Pipeline)
一个功能从产品概念到代码合入，必须严格经历以下五个阶段。任何阶段的产出物未通过自动化 Lint 校验，均不得进入下一阶段。

 阶段 ① [需求输入] ➡️ 人类编写高标准 PRD / 架构设计文档
      │
      ▼
 阶段 ② [拆解排产] ➡️ 拆单 AI 转换成带“依赖拓扑图”的 GitHub Issues 
      │
      ▼
 阶段 ③ [规范确立] ➡️ Spec Agent 在代码库生成机器可读的契约 (如 OpenAPI / SCHEMA.md) 
      │
      ▼
 阶段 ④ [双盲研发] ➡️ Test Agent (写测试) 与 Code Agent (写代码) 严格基于 Spec 并行开发
      │
      ▼
 阶段 ⑤ [闭环交付] ➡️ 脚手架自动运行回归测试，验证无偏离后，自动提交 PR 交由人类 Review

------------------------------
## 3. PRD 拆解与 Issue 规范 (PRD to Issues Standard)
为了让 AI 能够看懂需求并自动化执行，从 PRD 到 GitHub Issue 的拆解必须遵循以下标准化模板和技术指标。
## 3.1 Issue 拆单三大原则

   1. 原子性（Atomicity）：一个 Issue 只能包含一个核心功能点。严禁把“用户登录加支付功能”写进同一个 Issue。
   2. 契约前置（Contract-First）：Issue 的描述中不规定具体代码怎么写，但必须规定规范文件（Spec）更新的期望。
   3. 声明式依赖（Declarative Dependencies）：利用特定语法显式声明前置条件，供底层 Celery 调度器计算拓扑排序。

## 3.2 自动化 Issue 标准模板
每一个由 PRD 拆解出的 Issue，其 GitHub Body 必须严格满足以下 Markdown 结构：

<!-- GitHub Issue 自动化模板 -->
## 1. 任务声明 (Metadata)* **所属模块**: [例如：User / Auth / Payment]
* **依赖关系**: Depends on: #101  <!-- 【核心】供 Celery 脚本读取并做拓扑排产 -->
## 2. 规范契约要求 (Specification Requirements)* **影响的规范文件**: `docs/specs/api_v1.json` * **契约变更说明**: 
  - 新增 `/api/v1/payment/charge` 接口
  - 请求参数: `user_id` (string), `amount` (int, 单位分)
  - 期望返回: `200 OK` 伴随 `trade_no`
## 3. 验收测试标准 (Acceptance Criteria)* [ ] 正常流：当余额充足时，扣款成功并生成 trade_no。
* [ ] 异常流：当 amount <= 0 时，接口必须触发 400 报错。
* [ ] 边界值：高并发下同一个 user_id 的重复请求必须实现幂等。

## 4. 微观脚手架执行规范 (Scaffold Implementation Spec)
这是针对我们之前讨论的 Python + Celery 自动化脚手架制定的核心执行契约，确保 AI 在本地沙盒中安全、可预测地工作。
## 4.1 环境隔离 invariant（安全红线）

* 隔离性：脚手架必须为每一个 Issue ID 动态创建 git worktree，绝对不允许直接在主开发目录修改代码。
* 路径守卫：Agent 执行任何终端指令前，必须强校验当前路径 cwd == /tmp/air_workspaces/issue_id，防止 AI 越权格式化系统盘或污染公用环境。

## 4.2 双盲 TDD 研发闭环 (Feedback Loop)
在单个 Issue 的 Celery 任务链中，AI 严禁“既当裁判又当运动员”。

   1. Test Agent 阶段：脚手架启动 Copilot，仅读取 Issue 和现有的规范文件，在 tests/ 下编写自动化测试用例。
   2. 断链熔断（Red Light Gateway）：脚手架在本地运行 pytest。此时测试必须失败。如果测试直接通过了，证明 AI 编写了无效测试，Celery 必须立刻阻断任务。
   3. Code Agent 阶段：脚手架启动第二个 Copilot 上场。它不看原始需求，它只看新写好的测试用例 + 规范文件，其唯一目标就是通过修改 src/ 代码让红灯变绿。

------------------------------
## 5. 项目落地注意事项与防呆机制 (Operational Caveats)

   1. 防止 AI 串通修改规范（Spec Drift）：
   在开发阶段，Code Agent 可能会因为能力不足，偷偷去把 API_SPEC.json 里的校验规则改简单（例如把参数必填改成选填）来让自己写的烂代码通过测试。
   * 对策：脚手架在最终运行测试前，必须通过 git diff 检查。如果在 Code Agent 阶段规范文件被动过了，一律视为非法，直接报警熔断。
   2. 处理循环依赖（Circular Dependencies）：
   人工写 PRD 或 AI 拆单时，可能会不小心写出“Issue A 依赖 Issue B，Issue B 又依赖 Issue A”的死锁情况。
   * 对策：底层 Python 调度器在读取 GitHub Issues 列表时，必须首先通过 graphlib.TopologicalSorter 进行有向无环图（DAG）校验，一旦发现环，全盘拒绝排产，通知人工架构师介入修复。
   3. 人类 Review 的“黄金双向反馈”：
   当一切自动化通过，GitHub 上会多出一个包含 [Specification 变更 + Test Case + Source Code] 的高标准三合一 PR。人类架构师进行 Review 时，如果发现不对，不要直接改代码。
   * 正确做法：人类只需要在 GitHub Issue 评论区写下：“规范不符合预期，补充 XX 场景”。脚手架会自动捕捉评论，重新触发 Celery 链，让 AI 按照新提示重新迭代规范并重写代码。
