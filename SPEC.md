# AI + 人工协同开发规范

Status: Draft v1

## 1. 目的与适用范围

本规范用于统一本仓库的人机协同开发方式，约束从需求输入、规范沉淀、Issue 拆解、Agent 执行、测试验证到 PR 审查的完整流程。

本规范同时服务两类参与者：

- 人工角色：产品、架构、研发、审查者
- AI 角色：Issue 拆解助手、Copilot Coding Agent、自动化调度脚手架

本规范的目标不是替代现有运行时规范，而是把以下已有约束串成一套可执行的团队开发制度：

- `WORKFLOW.md`：Agent 运行时工作流与调度配置
- `.github/copilot-instructions.md`：仓库级编码与自动化执行规范
- `.github/ISSUE_TEMPLATE/*`：标准化提单模板
- `.github/pull_request_template.md`：PR 描述模板

## 2. 规范优先级

发生冲突时，按以下优先级解释与执行：

1. `.github/copilot-instructions.md`
2. `AGENTS.md`
3. `WORKFLOW.md`
4. 本文档
5. 一般性 README 与其他说明文档

说明：

- 编码、测试、提交流程的强制要求以 `.github/copilot-instructions.md` 为准。
- Agent 运行时行为、hooks、轮询、并发、工作区等以 `WORKFLOW.md` 为准。
- 本文档负责把这些要求翻译成团队日常协作规范与交付标准。

## 3. 核心原则

### 3.1 Spec First

需求进入研发前，必须先明确边界、验收标准和交付物。复杂需求应先形成设计说明、接口契约、流程说明或模块级技术规范，再拆解为 Issue。

### 3.2 Issue Is The Unit Of Execution

自动化脚手架和 Coding Agent 以单个 Issue 作为最小执行单元。Issue 必须做到可理解、可执行、可验证，不能依赖口头补充信息才能落地。

### 3.3 Human Decides, AI Executes

人负责方向、边界、取舍和最终审查；AI 负责阅读上下文、修改代码、编写测试、执行验证和准备 PR。不要把模糊决策留给 Agent 猜测。

### 3.4 Test And Evidence Driven

任何变更都必须有可验证证据。证据至少包括：

- 对应测试或等价验证
- 变更说明
- PR 模板中的测试计划

### 3.5 Isolated And Reproducible

自动化执行必须在隔离工作区中进行，避免污染主工作目录。涉及自动调度时，工作区隔离、路径守卫、失败断链属于安全红线。

## 4. 角色与职责

### 4.1 人工角色

`PM / 需求方`

- 提供 PRD、业务背景、成功标准和优先级
- 确认需求边界，不把相互独立的能力揉进一个 Issue

`架构师 / Tech Lead`

- 决定整体方案、模块边界和演进路线
- 审核规范、拆单策略与依赖关系
- 维护 `WORKFLOW.md`、`.github/copilot-instructions.md` 和关键模板

`研发工程师 / Reviewer`

- 审查 AI 产出的代码、测试和 PR 描述
- 关注正确性、安全性、可维护性和回归风险
- 对高风险改动给出结构化反馈，必要时回退到人工实现

### 4.2 AI 角色

`Issue 整理助手`

- 按 `.github/ISSUE_TEMPLATE` 生成标准化 Issue 草案
- 补齐缺失信息的最小草案，并明确待补充项

`Copilot Coding Agent`

- 读取 Issue、仓库规范与相关代码
- 实现代码、补测试、运行验证、准备 PR
- 在无阻塞前提下连续推进，不为常规动作频繁请求确认

`Automation Runner / 调度脚手架`

- 按 `WORKFLOW.md` 拉取候选任务并控制并发
- 为每个任务创建隔离工作区
- 运行 hooks、追踪状态、处理重试与失败报告

## 5. 标准输入产物

### 5.1 需求入口

允许的研发入口包括：

- PRD
- 架构或设计文档
- 缺陷报告
- 工程改造任务

当输入是 PRD 或设计文档时，不应直接交给 Agent 编码。必须先经过“规范化”和“Issue 拆解”两步。

### 5.2 规范化产物

复杂工作进入研发前，建议至少具备以下一种规范化产物：

- 接口契约：如 OpenAPI、JSON Schema、Markdown API 约定
- 模块设计：组件职责、数据流、边界和依赖
- 业务规则：状态机、校验规则、错误码、权限约束
- 非功能要求：性能、安全、兼容性、可观测性

如果暂时没有结构化契约，至少要在需求文档或 Issue 中明确：

- 影响范围
- 不在范围
- 验收标准
- 依赖关系

## 6. 从 PRD 到 Issue 的拆解规范

### 6.1 拆解原则

所有 Issue 拆解必须满足以下原则：

1. 原子性：一个 Issue 只承载一个明确目标，避免“大而全”的混合任务。
2. 可验证性：Issue 必须自带验收标准、完成标准或可复现步骤。
3. 有边界：必须写清楚做什么和不做什么。
4. 可排产：存在前置依赖时必须显式声明，不能靠隐含顺序执行。
5. 可自动化：信息应足够让 Agent 在不追问的情况下完成 80% 以上工作。

### 6.2 Issue 类型选择

按仓库模板选择类型：

- 新功能或功能增强：使用 `.github/ISSUE_TEMPLATE/feature.yml`
- 缺陷修复：使用 `.github/ISSUE_TEMPLATE/bug.yml`
- 重构、工程治理、测试补齐、依赖升级：使用 `.github/ISSUE_TEMPLATE/task.yml`

### 6.3 必填内容要求

`feature`

- 用“作为…我希望…以便…”描述用户故事
- 验收标准必须可测试，避免“体验更好”“支持更多场景”这类模糊描述
- 如有接口、模块、性能要求，应写入“技术规格”

`bug`

- 必须能复现，包含环境、期望行为、实际行为和关键日志
- 日志必须脱敏，禁止提交密钥、令牌、密码和用户隐私

`task`

- 必须明确任务边界，至少写清“需要做”和“不在范围”
- 完成标准不能只写“代码更优雅”，必须可验证

### 6.4 依赖关系约定

若 Issue 依赖其他 Issue，必须在正文中显式声明，例如：

```md
## 依赖关系
Depends on: #123
```

约束如下：

- 依赖关系由提单人或拆单 Agent 明确写出
- 不允许循环依赖
- 下游 Issue 不应默认重定义上游已经确认的规范

### 6.5 提单质量门槛

只有满足以下条件的 Issue 才应添加 `ai-ready`：

- 模板必填字段已完整填写
- 验收标准或完成标准可执行、可测试
- 边界和上下文足够清晰
- 无敏感信息泄露
- 前置依赖已声明或确认不存在

## 7. 单个 Issue 的推荐执行流水线

### 7.1 默认流水线

单个 Issue 默认按以下顺序推进：

1. 读取 Issue、`AGENTS.md`、`WORKFLOW.md` 与 `.github/copilot-instructions.md`
2. 定位相关代码与测试
3. 先补测试或先补最小验证用例
4. 修改实现代码
5. 运行针对性验证
6. 更新必要文档
7. 提交 commit
8. 推送分支并创建 PR

### 7.2 适用 SDD/TDD 的增强流水线

对复杂功能、协议变更、跨模块改造，推荐使用“规范驱动 + 测试驱动”的增强流程：

1. 先更新规范或设计说明
2. 基于规范编写测试
3. 先验证测试能正确失败
4. 再实现业务代码使测试通过
5. 最后运行完整回归验证

适用场景：

- 新 API 或接口契约变更
- 多模块联动改造
- 易产生幻觉或边界漂移的需求
- 需要多人和多个 Agent 协作的任务

### 7.3 何时必须停止自动推进

仅在以下情况允许中断并回到人工：

- 缺少权限、凭据或登录态
- 远端网络或仓库权限阻断执行
- 需求冲突且无法由现有上下文推断
- 继续执行会覆盖用户未提交改动
- 发现模板、规范或依赖关系存在明显错误

## 8. 自动化脚手架使用规范

### 8.1 调度范围

自动化调度仅处理已规范化且满足执行门槛的任务。调度器不负责替代需求澄清。

### 8.2 双模式定义

脚手架必须同时支持以下两种模式：

1. 人工驱动模式（Manual / Supervised）
	- 人工先指定基于哪个文档拆解和执行哪个 Issue。
	- Agent 在每个关键步骤接受人工辅助和监督。
	- 适用于高风险、需求未稳定、或需要强可解释过程的任务。

2. 自动化执行模式（Automated）
	- 由调度器自动轮询、排产和执行 Issue。
	- 自动化调度统一使用 Celery dispatcher + worker。
	- 适用于任务量大、依赖复杂、需要并发吞吐的场景。

模式边界要求：

- 两种模式必须共用同一套编码规范和质量门禁。
- 两种模式的入口命令、输入要求、产出物要清晰可区分。
- 同一 Issue 不应在两种模式下并发执行。

### 8.3 工作区隔离

脚手架执行时必须遵守以下约束：

- 一个 Issue 对应一个隔离工作区
- Agent 的工作目录必须位于该工作区内
- 不允许直接在主仓库目录中执行自动改码

### 8.4 失败处理

自动化任务失败时，至少要保留以下信息：

- 失败阶段
- 错误摘要
- 已执行的验证
- 建议的下一步动作

### 8.5 依赖任务排产

存在依赖关系的任务必须按依赖图顺序执行。对于无依赖的任务，可以并行；对于依赖同一父任务的子任务，只有在上游达到可用状态后才允许调度。

Celery 模式下，依赖关系建议采用 `Depends on: #123` 声明，并按 DAG 分层调度：

- 同层任务可并发执行
- 下一层必须等待上一层全部完成
- 任一层失败时停止后续依赖层投递

## 9. 编码与测试规范摘要

编码与测试的强制细节以 `.github/copilot-instructions.md` 为准，这里只提炼团队执行时必须记住的底线：

- 公共函数和方法必须有类型注解
- 公共 API 必须有 docstring
- 不允许裸露异常捕获
- 不使用 `print()` 作为生产代码日志
- 所有配置从环境变量或配置层获取
- 新增功能或修复必须配套测试
- 单测与集成测试分层放置
- 不绕过 lint、类型检查和测试门禁

## 10. PR 规范

所有自动化或人工提交的 PR，都必须使用 `.github/pull_request_template.md` 的完整结构。

强制要求：

- `背景` 必须说明原因并关联 Issue
- `变更摘要` 必须可让不熟悉上下文的审查者快速理解改动
- `变更详情` 只写高层变更点，不写流水账
- `测试计划` 必须明确哪些已执行、哪些未执行以及原因
- `审查重点` 必须指出风险点、复杂逻辑或需要重点确认的地方

禁止行为：

- 只写一句“已测试通过”
- 保留模板占位文本
- 没有 Issue 关联就直接提 PR

## 11. Review 规范

人工 Review 重点检查以下内容：

- 需求是否被正确实现，而不是仅仅“测试通过”
- 是否存在越界修改或顺手重构
- 是否引入安全、兼容性或性能风险
- 测试是否真正覆盖验收标准
- PR 描述是否足以支撑高质量审查

当审查不通过时，优先反馈到：

- Issue 需求不清：回到 Issue 或设计文档修正
- 规范不对：先改规范再迭代代码
- 实现不对：在 PR 或 Issue 中明确指出缺口和预期行为

## 12. 文档同步规则

出现以下变化时，必须同步更新相关文档：

- 工作流入口、默认执行策略、hooks 或调度方式变化：更新 `README.md` 或 `WORKFLOW.md`
- 编码规范、测试要求、执行边界变化：更新 `.github/copilot-instructions.md`
- PR 模板或提单模板变化：更新 `.github` 下对应模板
- 团队协作方式变化：更新本文档

## 13. 落地清单

当团队准备正式使用本规范时，建议按以下顺序落地：

1. 确认 `WORKFLOW.md` 与 `.github/copilot-instructions.md` 反映当前真实流程。
2. 用 `.github/prompts/create-issue-prompt.md` 和 Issue 模板统一提单方式。
3. 为复杂模块补齐设计说明或契约文档，再进入拆单。
4. 仅为满足门槛的 Issue 打上 `ai-ready`。
5. 强制使用 PR 模板并审查测试计划的真实性。
6. 定期回顾失败案例，修正模板、规范和脚手架。
